Pada percobaan saya, total queue yang terlihat adalah 4. Jumlah queue tersebut terjadi karena publisher mengirim message lebih cepat daripada subscriber memprosesnya. Dalam program ini, setiap kali publisher dijalankan, publisher mengirim 5 event ke queue `user_created`.

- Pada percobaan ini, saya menjalankan tiga subscriber secara bersamaan. Ketiga subscriber tersebut terhubung ke message broker RabbitMQ yang sama dan mendengarkan queue yang sama, yaitu `user_created`.
Ketika publisher dijalankan, publisher mengirim 5 event ke RabbitMQ. Karena ada tiga subscriber yang aktif, message dari queue tidak hanya diproses oleh satu subscriber saja, tetapi dibagikan ke beberapa subscriber. Hal ini membuat pemrosesan message menjadi lebih cepat dibandingkan ketika hanya ada satu subscriber.
Pada terminal subscriber, terlihat bahwa setiap subscriber menerima message yang berbeda-beda. Ini menunjukkan bahwa RabbitMQ membagi beban pemrosesan message ke beberapa consumer. Dengan begitu, jika ada banyak message yang masuk ke queue, queue dapat berkurang lebih cepat karena diproses secara paralel oleh beberapa subscriber.
Pada chart RabbitMQ, spike queue juga turun lebih cepat dibandingkan saat hanya ada satu slow subscriber. Hal ini terjadi karena jumlah consumer bertambah, sehingga kapasitas pemrosesan message juga meningkat. Percobaan ini menunjukkan salah satu kelebihan event-driven architecture, yaitu sistem dapat lebih mudah diskalakan dengan menambahkan subscriber atau consumer baru.

- Dari kode publisher dan subscriber, ada beberapa hal yang bisa diperbaiki. Pertama, struktur `UserCreatedEventMessage` saat ini didefinisikan secara terpisah di publisher dan subscriber. Akan lebih baik jika struktur event tersebut dipisahkan ke dalam library atau crate bersama, agar publisher dan subscriber menggunakan definisi message yang sama dan mengurangi duplikasi kode.
Kedua, proses publish pada publisher saat ini menggunakan `_ = p.publish_event(...)`, sehingga error yang mungkin terjadi tidak ditangani dengan jelas. Akan lebih baik jika setiap proses publish menggunakan error handling, misalnya dengan `match` atau `expect`, agar ketika pengiriman event gagal, penyebabnya lebih mudah diketahui.
Ketiga, subscriber menggunakan loop kosong `loop { }` untuk menjaga program tetap berjalan. Cara ini kurang efisien karena dapat menggunakan CPU tanpa melakukan pekerjaan yang berarti. Akan lebih baik jika menggunakan mekanisme blocking atau waiting yang lebih tepat.
Kesimpulannya, menjalankan tiga subscriber membuat proses konsumsi message menjadi lebih cepat karena message diproses secara paralel. Ini membuktikan bahwa event-driven architecture dapat membantu sistem menjadi lebih scalable dan lebih tahan terhadap beban tinggi.
